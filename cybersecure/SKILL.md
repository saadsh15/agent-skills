# /cybersecure

## Overview
The `cybersecure` skill is a remediation and hardening agent designed to work in tandem with the `/cybercheck` skill. Its primary purpose is to ingest the `CYBERCHECK_REPORT.md` generated by an audit, deeply analyze the root causes of the vulnerabilities, and implement robust security fixes. Crucially, `cybersecure` focuses on **safe remediation**—ensuring that security patches, rate limits, and structural changes do not break existing functionality, especially for public-facing REST APIs and critical user flows.

## Instructions

When the user invokes `/cybersecure`, you must execute the following workflow:

### 1. Ingestion & Root Cause Analysis
- Locate and read the `CYBERCHECK_REPORT.md` in the root of the project.
- For each vulnerability listed, perform a **Root Cause Analysis (RCA)**:
  - *Why* is the application weak here? (e.g., lack of framework-level middleware, legacy code patterns, misconfigured ORM).
  - Determine the "blast radius" of fixing this issue (e.g., will changing this API payload structure break mobile clients?).

### 2. Safe Remediation Strategy Formulation
Before writing any code, formulate a strategy that guarantees zero (or minimal) breakage:
- **API Compatibility:** If securing an endpoint requires changing its signature or expected headers (e.g., adding CSRF tokens or requiring new Auth headers), determine if a versioning strategy (e.g., `/v2/`) or a transitional deprecation phase is needed.
- **Stateful Changes:** If fixing database injections requires migrating from raw queries to an ORM/Parameterized queries, ensure the schema and return types remain strictly identical.
- **Rate Limiting (DDoS/DoS):** When applying rate limiting, analyze typical legitimate traffic spikes to set sensible thresholds (e.g., burst allowance) so legitimate users or microservices aren't locked out.

### 3. Execution & Implementation Policies
Apply the following specialized solutions to the generalized use cases found in the report:

#### A. APIs (REST, GraphQL, gRPC) & Web Backends
- **Vulnerability:** SQL/NoSQL Injection.
  - **Specialized Solution:** Refactor raw queries into parameterized queries or use the project's existing ORM safely. Do not alter the JSON response shape.
- **Vulnerability:** XSS (Cross-Site Scripting).
  - **Specialized Solution:** Implement context-aware output encoding. For APIs returning JSON, ensure proper `Content-Type: application/json` headers are strictly enforced to prevent browsers from sniffing and executing malicious payloads.
- **Vulnerability:** Broken Authentication / Insecure JWTs.
  - **Specialized Solution:** Upgrade token signing algorithms (e.g., RS256). If rotating secrets is required, implement a graceful rollover mechanism where both old (for a brief window) and new keys are accepted to prevent active sessions from dropping abruptly.
- **Vulnerability:** DDoS / Lack of Rate Limiting.
  - **Specialized Solution:** Implement sliding-window rate limiting via Redis (or in-memory if single-instance). Add `X-RateLimit-*` headers so well-behaved clients can adapt gracefully.

#### B. Frontend Web Applications
- **Vulnerability:** DOM-based XSS.
  - **Specialized Solution:** Sanitize user input using established libraries (e.g., DOMPurify) before injecting into the DOM (e.g., before `innerHTML`). Avoid breaking rich-text editors by carefully configuring allowed tags.
- **Vulnerability:** CSRF.
  - **Specialized Solution:** Implement the Double Submit Cookie pattern or `SameSite=Strict`/`Lax` cookie attributes. If adding CSRF tokens breaks SPA (Single Page Application) state, implement an endpoint for the frontend to fetch the token dynamically on initialization.

#### C. Databases & Infrastructure
- **Vulnerability:** Overly permissive database roles.
  - **Specialized Solution:** Implement the Principle of Least Privilege. Create separate database users for read-only operations vs. write operations and update the application's connection pools accordingly.
- **Vulnerability:** Exposed Secrets.
  - **Specialized Solution:** Migrate hardcoded secrets to environment variables or a Secret Manager (e.g., AWS Secrets Manager, HashiCorp Vault). Provide a `.env.example` file to ensure the development environment remains functional.

### 4. Breakage Management & Fallback Plans
If applying a security policy breaks the application or test suite, follow this recovery protocol:
1. **Identify the Breakage:** Analyze the stack trace or failing test to determine the exact conflict (e.g., strict CSP blocking a legitimate third-party analytics script).
2. **Graceful Degradation:** Instead of completely removing the security control, adjust its strictness. 
   - *Example:* If a strict Content Security Policy (CSP) breaks the site, switch it to `Content-Security-Policy-Report-Only` and log the violations to identify what needs to be allowlisted.
   - *Example:* If rate limiting blocks legitimate webhooks, implement an IP allowlist bypass for known internal/partner IPs.
3. **Isolate and Revert:** If the fix fundamentally breaks core business logic and cannot be tuned, revert the specific change and document the limitation in a `SECURITY_DEBT.md` file with a proposed long-term architectural fix.

### 5. Final Documentation
Once the fixes are applied and verified against the existing project tests, generate a `CYBERSECURE_REMEDIATION.md` report detailing:
- Which vulnerabilities from the `/cybercheck` report were fixed.
- The exact code changes made.
- Any architectural compromises made to preserve functionality.
- Instructions for the development team on how to maintain these new security measures moving forward.
